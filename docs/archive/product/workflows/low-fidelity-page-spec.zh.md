# KubeDeck 首页与 Workloads 页低保真页面规范

## 1. 目的

本文档将当前的产品需求、任务流草案、状态规则和字段级信息架构，进一步收敛为首页与 Workloads 页的低保真页面规范。

目标不是视觉设计打磨，而是定义用户首先应该看到什么、每个区域必须承担什么职责，以及页面如何支撑首个核心任务流。

## 2. 范围

本低保真规范覆盖：
- 首页
- Workloads 页面

本规范暂不定义：
- 最终视觉风格，
- 最终组件选型，
- 响应式像素级布局，
- 或高保真动效细节。

## 3. 首页低保真规范

### 3.1 页面目标

首页必须：
- 建立当前操作上下文，
- 提供进入首条任务流的主入口，
- 并避免用大量诊断信息与该任务竞争注意力。

### 3.2 页面区域

首页应按以下顺序包含这些顶层区域：

1. 全局头部
2. 当前上下文区域
3. 主任务入口区域
4. 恢复 / 默认任务区域
5. 阻塞性提示区域

### 3.3 线框级结构

```text
+--------------------------------------------------------------+
| 头部：产品 / 会话 / 次级设置                                  |
+--------------------------------------------------------------+
| 当前上下文                                                     |
| Cluster: [prod v]   Namespace: default / last used           |
+--------------------------------------------------------------+
| 主任务入口                                                     |
| Workloads                                                     |
| 在当前 cluster 中浏览并操作 workloads                         |
| [进入 Workloads]                                               |
+--------------------------------------------------------------+
| 恢复或默认任务                                                 |
| 恢复上次上下文，或进入默认任务入口                             |
+--------------------------------------------------------------+
| 阻塞性提示（仅在需要时出现）                                   |
| 问题摘要 / 下一步动作 / 重试                                   |
+--------------------------------------------------------------+
```

### 3.4 区域职责

#### 全局头部

必须做到：
- 标识产品，
- 提供最小必要的全局入口，
- 在视觉上从属于任务入口主链路。

不能做到：
- 承载任务相关诊断信息，
- 把 cluster 级运行细节作为主信息表达。

#### 当前上下文区域

必须做到：
- 展示当前 `activeCluster`，
- 允许用户切换 cluster，
- 展示进入工作前所需的最小 namespace 上下文。

不能做到：
- 让用户承受过多 cluster 深层元数据，
- 让技术标识成为主内容。

#### 主任务入口区域

必须做到：
- 让 `Workloads` 成为最明确的下一步，
- 用一句短说明解释进入后能做什么，
- 提供单一清晰动作。

不能做到：
- 让多个入口在同一优先级竞争，
- 迫使用户理解技术来源分类。

#### 恢复 / 默认任务区域

必须做到：
- 帮助回访用户继续最可能的下一步，
- 在没有最近上下文时提供默认入口。

不能做到：
- 压过主任务入口，
- 演变成历史活动面板。

#### 阻塞性提示区域

必须做到：
- 仅在首条任务流确实被阻塞时出现，
- 概括阻塞原因和下一步动作。

不能做到：
- 展示非阻塞性的壳层诊断，
- 在没有真正阻塞时长期占据主内容区域。

### 3.5 首页验收条件

当满足以下条件时，可认为首页达标：
- 用户能立刻识别当前 cluster，
- 用户能无歧义进入 `Workloads`，
- 任务入口比诊断信息更突出，
- 基础使用不依赖按来源分类的导航。

## 4. Workloads 页低保真规范

### 4.1 页面目标

Workloads 页必须：
- 成为主操作工作面，
- 明确展示 cluster 与 namespace 上下文，
- 并让用户快速开始一次标准操作。

### 4.2 页面区域

Workloads 页应按以下顺序包含这些顶层区域：

1. 页面头部
2. 上下文与范围栏
3. 过滤与搜索栏
4. 主操作区域
5. Workload 列表区域
6. 结果 / 状态区域

### 4.3 线框级结构

```text
+--------------------------------------------------------------+
| 头部：Workloads                            Cluster: prod     |
+--------------------------------------------------------------+
| 范围栏                                                         |
| Namespace: [default v]   Filters: Running, Error   [Clear]   |
+--------------------------------------------------------------+
| 搜索 / 筛选                                                    |
| [Search........] [Status v] [Type v]                           |
+--------------------------------------------------------------+
| 操作区                                                         |
| [Create] [Apply] [Refresh]                                     |
+--------------------------------------------------------------+
| Workload 列表                                                  |
| Name        Kind        Namespace      Status      Health      |
| api         Deployment  default        Running     Ready       |
| web         Deployment  default        Pending     Warning     |
+--------------------------------------------------------------+
| 结果 / 状态（仅在有上下文意义时展示）                           |
| 最近操作成功 / 部分失败 / 失败                                 |
+--------------------------------------------------------------+
```

### 4.4 区域职责

#### 页面头部

必须做到：
- 明确用户当前位于 `Workloads`，
- 清晰重申当前 cluster。

不能做到：
- 承载大量次级说明，
- 演变成仪表盘摘要面板。

#### 上下文与范围栏

必须做到：
- 明确 namespace 范围，
- 展示当前筛选状态，
- 支持快速修正上下文。

不能做到：
- 将 namespace 藏到深层交互后，
- 把上下文分散到多个无关区域。

#### 搜索与筛选栏

必须做到：
- 支持快速收缩可见资源范围，
- 暴露首条任务流最相关的一到两个筛选条件。

不能做到：
- 在基础流程尚未成立前，就用高级筛选增加复杂度。

#### 主操作区域

必须做到：
- 将 `Create` 和 `Apply` 暴露为首要操作入口，
- 让用户一眼知道能做什么动作。

不能做到：
- 将主操作藏入行级菜单或次级控件中。

#### Workload 列表区域

必须做到：
- 展示真实资源列表，
- 提供足够识别信息，帮助用户对正确对象操作，
- 清楚展示当前状态。

不能做到：
- 首先为指标、图表、概览卡片优化，
- 要求用户先钻取详情页才能知道当前有哪些对象。

#### 结果 / 状态区域

必须做到：
- 在需要时展示最近操作结果，
- 并把结果与当前列表上下文关联起来。

不能做到：
- 在没有操作发生时长期占据黄金区域，
- 展示与当前 workload 任务无关的壳层诊断。

### 4.5 Workloads 页验收条件

当满足以下条件时，可认为 Workloads 页达标：
- 用户能立刻识别当前 cluster 和 namespace，
- 用户能看到真实资源列表，
- 用户无需查找即可开始 `Create` 或 `Apply`，
- 用户无需离开当前任务路径就能理解操作结果。

## 5. 跨页面一致性规则

这两页都必须遵守以下规则：
- cluster 上下文在页面入口处始终可见，
- 按来源分类的导航标签不能成为主 UX 语言，
- 除非阻塞当前任务，否则诊断信息必须降级，
- 页面必须先回答“我接下来做什么”，再回答“这个壳怎么实现”。

## 6. 下一层规范待决问题

下一份文档需要继续明确：
- `Create / Apply` 到底是页面、抽屉还是弹窗，
- 首页与 Workloads 之间 namespace 如何持久化，
- 首版 workload 列表精确显示哪些列或卡片，
- 以及结果反馈如何在视觉上挂接到当前任务流中。
