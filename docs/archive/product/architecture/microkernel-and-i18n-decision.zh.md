# KubeDeck 微内核扩展架构与 i18n 最低约束

## 1. 目的

本文档用于明确：在当前阶段，微内核架构与 i18n 分别有哪些内容必须立即进入产品架构，哪些内容可以在首条工作流稳定后再做。

目标是避免两类问题：
- 为了尽快做出首条工作流，把内置功能继续硬编码进壳层；
- 或把 i18n 完全推迟，导致后续不得不进行一轮机械性重写。

## 2. 决策结论

这两个问题的重要程度并不相同。

微内核扩展能力属于产品定义级架构问题，必须现在进入实现主线。

i18n 同样重要，但完整 i18n 能力可以后置。现在必须进入架构的是一组最低约束，用来防止后续页面增长时把文案和行为继续绑死在一起。

## 3. 为什么微内核必须现在进入架构

KubeDeck 的产品定义是“多集群、可插件扩展的控制平面”，而不是一个固定内置功能的控制台。

这意味着以下问题都不是次要实现细节：
- 页面如何注册，
- 菜单如何贡献，
- 动作如何挂接到工作流，
- 内置资源域如何表达，
- 后端能力如何暴露给壳层。

如果这些问题继续后置，首条工作流很容易被实现成一组应用内特例。那样后续再做“插件化”时，就不再是平滑演进，而会变成结构性回拆。

## 4. 为什么完整 i18n 可以后置

当前阶段，首条真实工作流仍然比完整多语言交付更重要。

如果完整 i18n 会拖慢首条可用控制面工作流，KubeDeck 的 V1 不需要先把它全部做完。

但这并不意味着项目可以继续无约束地新增硬编码页面文案。若继续这样做，后续 i18n 就会变成覆盖壳层、页面和插件合同的一轮大规模机械替换。

## 5. 必须现在进入架构的微内核约束

以下微内核约束必须立即进入架构主线。

### 5.1 内置功能也必须遵守内核规则

像 `Homepage`、`Workloads` 以及对应动作这类内置能力，不能继续被当成永久性的壳层特例。

它们可以先随主仓库一起交付，但其建模方式应当是“一等内核贡献”，而不是临时写死的应用路径。

### 5.2 前端必须有明确的贡献合同

前端壳层必须显式定义至少以下几类稳定合同：
- 页面注册，
- 菜单贡献，
- 动作贡献，
- 扩展槽位贡献。

壳层仍然可以提供默认内置实现，但内置能力也应尽可能通过同一组合同接入。

### 5.3 后端必须有明确的能力边界

后端内核必须定义至少以下几类稳定边界：
- 插件身份，
- 资源或工作流能力注册，
- 菜单或页面元数据暴露，
- 以及带权限约束的执行入口。

后端不能只停留在“stub 路由 + 很薄的插件注册器”这一层。

### 5.4 首条工作流必须与扩展兼容

首条工作流现在不需要马上支持完整第三方插件交付，但其结构不能阻碍未来：
- 增加新的工作流域，
- 在现有工作流中贡献动作，
- 挂接侧边面板或上下文扩展，
- 在不重写壳层的前提下扩展内置资源处理。

### 5.5 必须明确内核归属

内核负责：
- 贡献注册，
- 组合规则，
- 路由与菜单装配，
- 工作上下文边界，
- 以及带权限意识的执行边界。

插件或内置能力模块负责：
- 自己的页面，
- 自己的动作，
- 自己的领域视图逻辑，
- 以及自身的贡献元数据。

## 6. 可以后置的微内核能力

以下微内核相关能力可以在首条工作流稳定后再做：
- 运行时动态加载第三方插件，
- 插件热安装与卸载，
- 丰富的插件管理或市场界面，
- 覆盖所有页面区域的完整 slot replacement 语义，
- 以及覆盖多个资源域的大规模插件化接入。

这些方向是合理的长期演进方向，但不是当前确定内核形状所必需的条件。

## 7. 必须现在进入架构的 i18n 最低约束

即使完整 i18n 后置，以下最低规则也应从现在开始执行。

### 7.1 新增 UI 文案必须有 locale 边界

新的产品 UI 文案不应继续以随意硬编码字符串的方式散落在页面和 feature 组件中。

至少要建立一种统一的文案访问方式或存放位置，即使当前只填充一个 locale，也要先把边界立住。

### 7.2 合同必须支持可本地化标签

菜单、页面、动作、扩展贡献等合同，应从现在起就允许标题和描述未来可被本地化，而不是默认永远只有一个固定展示字符串。

### 7.3 locale 必须被视为产品级偏好

即使短期内还没有语言切换 UI，locale 也应被视为产品级偏好，而不是页面局部状态。

### 7.4 文案与逻辑必须保持分离

实现时应避免把控制逻辑和用户可见文案混写到难以抽离的程度。

这条规则尤其适用于：
- 页面标题，
- 按钮文本，
- 结果反馈，
- 空状态文案，
- 工作流引导文案。

## 8. 可以后置的 i18n 能力

以下内容可以在首条工作流稳定后再做：
- 完整的语言切换 UI，
- 完整的 EN/ZH 运行时切换，
- 日期、时间、数字的全面本地化格式化，
- 后端持久化 locale 偏好，
- 以及第三方插件自带的翻译包体系。

这些内容很重要，但不应阻塞首条可用工作流。

## 9. 当前立即执行规则

从现在开始：
- 新的实现工作不能继续加深“壳层专属内置特例”；
- 新的工作流界面要先经过内核贡献边界审视；
- 新增 UI 文案不能在没有未来可本地化路径的情况下直接写死。

一句话总结：
- 微内核形状是“现在决策”；
- 完整 i18n 交付是“后续决策”；
- i18n 最低纪律也是“现在决策”。

## 10. 架构就绪检查

在继续进行大规模实现前，以下问题必须可以回答“是”：
- 内置工作流页面是否可以被描述为内核贡献，而不是壳层特例？
- 菜单、页面、动作、扩展边界是否已经显式化？
- 首条工作流是否为后续插件贡献留出了空间？
- 新增用户文案未来是否可以在不重写组件逻辑的前提下完成本地化？

如果任一答案为“否”，就应先补回架构工作，再继续扩大功能实现。
