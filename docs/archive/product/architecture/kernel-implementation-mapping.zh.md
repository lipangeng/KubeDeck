# KubeDeck 内核实现映射草案

## 1. 目的

本文档在不依赖现有历史代码布局的前提下，把内核贡献合同进一步映射成面向实现的职责区域。

## 2. 映射原则

内核应保持小、稳、以组合为中心。

能力模块应承载工作流特定代码。

这份映射描述的是目标职责归属，而不是历史文件保留策略。

## 3. 前端职责区域

### 3.1 内核壳层

前端内核壳层应负责：
- 贡献注册
- 菜单组合
- 工作流路由入口解析
- 共享工作上下文访问
- 槽位渲染编排

壳层不应负责：
- 内置工作流页面的详细渲染
- 动作表单的业务逻辑

### 3.2 内置能力模块

内置能力模块应负责：
- `Homepage` 页面贡献
- `Workloads` 页面贡献
- `Create` 动作贡献
- `Apply` 动作贡献

这些模块在 V1 中虽然属于内置能力，但其行为方式应已经像能力贡献者，而不是壳层私有代码。

### 3.3 共享上下文层

共享上下文层应保持贴近内核，负责：
- cluster 连续性
- namespace 连续性
- workflow 连续性
- list 连续性
- action 连续性

能力模块可以消费它，但不能重定义它。

### 3.4 槽位层

槽位层应负责：
- 槽位定义
- 槽位查找与渲染规则
- 无贡献时的安全回退行为

V1 可以先保持很窄，但职责边界现在就要存在。

## 4. 后端职责区域

### 4.1 后端内核

后端内核应负责：
- 能力注册
- 能力元数据组合
- cluster 感知的元数据解析
- 后端权威执行入口
- 权限与策略执行

### 4.2 内置能力提供者

内置提供者应负责：
- workload 元数据贡献
- workload 页面能力元数据
- create/apply 动作能力元数据
- 内核执行入口之后的资源操作细节

### 4.3 registry 与元数据层

registry 与元数据层应负责：
- 能力发现结构
- cluster 作用域下的菜单/页面/动作可见性解析
- 面向前端的安全传输元数据载荷

### 4.4 执行层

执行层应负责：
- 动作派发
- 目标校验
- 权限校验接入
- 结果摘要生成

## 5. V1 推荐模块分组

在继续扩大实现前，项目至少应建立以下概念分组：

- frontend kernel shell
- frontend built-in capability modules
- frontend shared context layer
- frontend slot layer
- backend kernel registry/composition layer
- backend built-in capability providers
- backend execution layer

## 6. 不应映射进内核的内容

以下内容不应被当成内核职责：
- 页面局部表单状态
- workload 表格展示细节
- 诊断面板布局
- AI 助手逻辑
- 插件市场行为

这些内容未来可以消费内核合同，但它们本身不属于内核归属。

## 7. V1 映射就绪检查

只有当以下条件成立时，才适合继续实现：
- 内置工作流代码已经有清晰的归属位置，不再留在壳层核心中
- 共享工作上下文仍然独立于能力模块
- 后端执行仍然独立于菜单/页面元数据组合
- 槽位归属已定义，即使首批槽位非常少
