# KubeDeck i18n 运行时模型草案

## 1. 目的

本文档定义一组最低运行时模型，使 KubeDeck 在继续增长时，不会阻塞未来的本地化工作。

它不要求 V1 先完整交付 i18n。

## 2. 运行时原则

V1 可以只以一个主要 runtime locale 交付，但运行时模型必须先把以下几层与页面逻辑分离：
- locale 偏好
- 文案访问方式
- 可本地化贡献元数据

## 3. 最低运行时要素

### 3.1 Locale 偏好

Locale 应被视为产品级偏好状态。

它不应被视为页面局部 UI 状态。

### 3.2 文案访问层

用户可见文本应通过统一文案访问方式取得，而不是继续以随意内联字符串散落在页面和 feature 组件中。

V1 可以先保持简单，但访问路径必须先存在。

### 3.3 可本地化贡献元数据

内核贡献从现在起就应允许标题和描述未来可被本地化。

V1 即使仍然只从一个 locale 源解析，合同形状也必须允许后续扩展。

### 3.4 格式化边界

日期、时间、数字等与 locale 相关的展示，应被视为“格式化边界”，而不是继续做普通字符串拼接。

即使完整格式化覆盖后置，这条边界也必须先被承认。

## 4. V1 必须做到的事

V1 应做到：
- 把 locale 视为产品级偏好概念
- 避免继续扩散内联文案
- 让贡献元数据具备未来可本地化能力
- 在新增代码中保持文案与控制逻辑分离

## 5. V1 不需要先做到的事

V1 不需要先交付：
- 完整运行时语言切换
- 完整翻译覆盖
- 全面 locale 感知格式化
- 后端 locale 持久化

## 6. 新代码规则

从现在开始，新增产品 UI 文案都应以一种未来可以迁移到 locale lookup 后面的方式接入，而不是继续依赖页面重写。

这条规则适用于：
- 页面标题
- 按钮文本
- 动作标签
- 结果反馈
- 空状态文案
- 工作流提示文案

## 7. 运行时就绪检查

在继续扩大 UI 之前，以下问题必须可以回答“是”：
- Locale 是否已经被视为产品级偏好概念？
- 新增 UI 文案未来是否可以在不重写页面逻辑的情况下迁移到 locale lookup？
- 贡献标签未来是否可以在不改合同形状的情况下完成本地化？
