# KubeDeck V1 首批开发任务

## 1. 目的

本文档将 V1 实施任务拆分进一步收敛为首批可执行开发任务。

目标是定义一个最小但可落地的起始集合，让产品从文档阶段进入实现阶段，同时不重新打开范围。

## 2. 选择规则

首批任务应当：
- 解锁共享工作流基础，
- 避免视觉打磨工作，
- 在核心上下文规则稳定前避免过早并行，
- 并为 Homepage -> Workloads -> Create / Apply 的连续性打下最低基础。

## 3. 首批范围

首批任务只应包含 Phase 1 以及 Phase 2 的最小入口部分：

1. 共享工作上下文模型
2. cluster 切换生命周期
3. namespace 上下文基线
4. Homepage 主任务入口交接

在这四项稳定前，其他内容都应等待。

## 4. 首批开发任务

### 任务 A：共享工作上下文模型

目标：
- 为首条任务流定义并实现一套共享状态模型。

必须包含：
- active cluster
- namespace scope
- 当前资源域
- 页面间任务连续性所需字段

验收：
- 存在一套规范的上下文状态模型
- Homepage 与 Workloads 可依赖同一上下文来源
- 上下文字段不会无理由散落复制到无关本地状态中

为什么先做：
- 这是后续页面和操作的基础契约

### 任务 B：Cluster 切换生命周期

目标：
- 在 UI 流程扩张前，先让 cluster 切换变得可预测且安全。

必须包含：
- 切换时的加载状态
- 失败时的安全回退
- 防止失败后旧上下文和新上下文混杂

验收：
- cluster 切换要么干净完成，要么安全回退
- 切换后的上下文不存在歧义
- 失败不会让用户落入破损任务路径

为什么第二个做：
- namespace 和任务入口都依赖稳定的 cluster 行为

### 任务 C：Namespace 上下文基线

目标：
- 实现 V1 所需的 namespace 连续性行为。

必须包含：
- `single` 浏览范围
- `all` 浏览范围
- Homepage 到 Workloads 的继承
- 操作时收敛为明确执行目标

验收：
- 浏览语义与操作语义不混淆
- create/apply 不能针对模糊 namespace 目标提交
- namespace 上下文能贯穿首条任务流

为什么第三个做：
- 除 cluster 之外，这是风险最高的上下文规则

### 任务 D：Homepage 主任务入口

目标：
- 让 Homepage 成为首条任务流的真实入口。

必须包含：
- active cluster 清晰展示
- 直接进入 `Workloads`
- 壳层诊断从主视觉焦点降级

验收：
- Homepage 能清楚展示当前 cluster
- 用户能直接进入 `Workloads`
- 首屏由任务主导，而不是由诊断主导

为什么第四个做：
- 当前文规则稳定后，Homepage 才能真正成为工作流起点

## 5. 第二批预览

只有首批任务稳定后，才应进入下一批：
- Workloads 页面壳
- 真实 workload 列表
- Create / Apply 操作界面

## 6. 首批明确不做

首批任务不应包含：
- AI
- 插件工作流扩张
- 细节视觉打磨
- 高级列表筛选
- 最近任务 / 恢复任务 UX
- `multiple` namespace 的 UI
- workload 详情页扩张

## 7. 开始前检查清单

只有满足以下条件，首批任务才算可以开始：
- 共享上下文字段已达成一致
- namespace 规则已被接受
- Homepage 与 Workloads 已确认是首条路径
- 本批次没有额外加入新的 V1 范围
