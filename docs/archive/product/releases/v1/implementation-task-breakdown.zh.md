# KubeDeck V1 实施任务拆分

## 1. 目的

本文档将 V1 实现边界拆解为可执行的实施顺序。

目标是定义出让首条真实工作流端到端可用所需的最小实现任务集合。

## 2. 执行原则

任务顺序应服从工作流依赖，而不是服从技术上的整洁感。

这意味着：
- 先上下文，后打磨，
- 先任务入口，后次级导航，
- 先真实数据流，后壳层增强，
- 先可用反馈，后功能扩张。

## 3. 阶段总览

V1 应按以下阶段交付：

1. 共享上下文基础
2. 首页任务入口
3. Workloads 工作面
4. Create / Apply 流程
5. 结果反馈与返回链路
6. 加固与验收验证

## 4. 任务拆分

### 阶段 1：共享上下文基础

#### 任务 1.1 定义共享工作上下文模型

必须覆盖：
- active cluster
- namespace scope
- 当前资源域
- 为连续性所需的基础列表筛选条件

输出：
- 一套对跨页面连续性达成共识的状态模型

为什么先做：
- 后续所有页面和操作都依赖上下文连续性

#### 任务 1.2 实现 cluster 切换生命周期规则

必须覆盖：
- cluster 切换时的加载行为
- 失败时的安全回退
- 切换失败后不能留下混合上下文

输出：
- 可预测的 cluster 切换行为

#### 任务 1.3 实现 namespace 上下文基线

必须覆盖：
- `single` namespace 浏览
- `all` namespace 浏览
- Homepage 到 Workloads 的继承
- create/apply 的目标收敛规则

输出：
- V1 可工作的 namespace 行为

### 阶段 2：首页任务入口

#### 任务 2.1 重构首页信息优先级

必须覆盖：
- active cluster 的清晰展示
- 直接的主任务入口
- 壳层诊断信息的降级

输出：
- 能清楚把用户送入首条任务流的首页

#### 任务 2.2 实现 Homepage -> Workloads 跳转

必须覆盖：
- 直接任务入口
- 进入 Workloads 时的上下文交接

输出：
- 进入主工作面的真实入口路径

### 阶段 3：Workloads 工作面

#### 任务 3.1 实现 Workloads 页面壳

必须覆盖：
- 页面头部
- 上下文与范围栏
- 搜索/筛选基线
- 主操作区域

输出：
- 第一块真实任务工作面

#### 任务 3.2 实现真实 workload 列表

必须覆盖：
- 列表数据加载
- 资源身份可见
- 状态可见
- 空状态与错误状态

输出：
- 可用于浏览的 workload 列表

#### 任务 3.3 在 Workloads 内保留上下文

必须覆盖：
- namespace 持续性
- 列表筛选条件连续性
- 操作后稳定返回目标

输出：
- workload 上下文不会被无谓重置

### 阶段 4：Create / Apply 流程

#### 任务 4.1 决定并实现 Create / Apply 承载界面

必须覆盖：
- 所选交互形式
- 界面内的上下文可见性
- 输入区

输出：
- 与 Workloads 连接起来的操作入口

#### 任务 4.2 实现校验与目标收敛

必须覆盖：
- namespace 目标收敛
- namespaced 与 cluster-scoped 区分
- 提交前校验

输出：
- 安全的提交前置条件

#### 任务 4.3 实现提交行为

必须覆盖：
- 提交流程生命周期
- 失败时保留输入
- 如果支持多文档 apply，则进行逐文档处理

输出：
- 真实可用的 create/apply 执行路径

### 阶段 5：结果反馈与返回链路

#### 任务 5.1 实现成功反馈

必须覆盖：
- 受影响对象摘要
- 返回引导
- 保留工作流上下文

输出：
- 用户能立即理解成功结果

#### 任务 5.2 实现失败与部分失败反馈

必须覆盖：
- 明确失败位置
- 保留输入
- 提供重试或修改路径

输出：
- 用户在失败后无需重头开始

#### 任务 5.3 返回 Workloads 并保留上下文

必须覆盖：
- 操作后的刷新
- cluster 与 namespace 范围保留
- 稳定的返回落点

输出：
- 端到端任务连续性

### 阶段 6：加固与验收验证

#### 任务 6.1 按 V1 边界清单进行验证

必须覆盖：
- 首页直接入口
- 上下文可见性
- 真实操作完成
- 反馈清晰度
- 返回后的连续性

输出：
- 相对于 V1 定义的显式通过/失败结论

#### 任务 6.2 移除或降级超范围干扰项

必须覆盖：
- 仍然压过任务面的诊断信息
- 干扰主工作流的未完成次级 UX

输出：
- 更紧的 V1 范围和更清晰的产品焦点

## 5. 本拆分明确不含内容

本任务拆分不包含：
- AI 集成
- 大范围插件工作流交付
- `multiple` namespace 的高级 UI
- 次级仪表盘
- 大范围多资源域导航扩张
- 非任务完成所必需的视觉打磨

## 6. 推荐的初始交付顺序

如果任务需要严格顺序执行，建议顺序为：

1. 共享上下文模型
2. cluster 切换规则
3. namespace 上下文基线
4. 首页任务入口
5. Workloads 页面壳
6. workload 列表
7. Create / Apply 操作界面
8. 校验与目标收敛
9. 提交行为
10. 成功/失败反馈
11. 返回链路
12. 验收验证

## 7. 可进入实现的就绪定义

只有满足以下条件，任务才算进入实现阶段：
- 它直接绑定首条任务流，
- 它的上下文依赖清晰，
- 它的成功条件可观察，
- 并且它不会静默扩大 V1 范围。
