# KubeDeck 首版实现边界清单

## 1. 目的

本文档定义 KubeDeck 首版实现必须包含什么，以及必须明确延期什么。

目标是在首个核心任务流真正可用之前，阻止范围继续膨胀。

## 2. 边界原则

只有当一条真实用户任务流能够端到端可用时，V1 才算完成。

V1 不能因为以下事实就被视为完成：
- 壳层已经渲染出来，
- metadata 接口已经存在，
- 或架构文档中已经声明了扩展点。

## 3. V1 必须交付

### 3.1 核心任务流

V1 必须交付一条完整工作流：

1. 进入首页
2. 确认或切换 cluster
3. 进入 `Workloads`
4. 查看真实 workload 列表
5. 发起 `Create` 或 `Apply`
6. 在目标收敛正确的前提下提交
7. 获得清晰结果反馈
8. 在不丢失状态的前提下回到 workload 上下文

### 3.2 必需页面

V1 必须包含：
- 首页
- Workloads 页面
- Create / Apply 操作界面
- 同一任务流中的上下文结果反馈

### 3.3 必需上下文

V1 必须保留：
- active cluster
- namespace 上下文
- 当前资源域
- 为任务连续性所需的关键列表筛选条件

### 3.4 必需 Namespace 支持

V1 必须支持：
- `single` namespace 范围，用于浏览与默认操作落点
- `all` namespace 范围，用于浏览

同时 V1 还必须保证：
- create/apply 在需要时收敛到具体 namespace 目标
- cluster-scoped 资源被明确处理
- namespace 上下文在 Homepage、Workloads 和 Create / Apply 之间保持连续

### 3.5 必需反馈

V1 必须提供：
- 包含受影响对象的成功反馈
- 保留输入上下文的失败反馈
- 当存在多文档或混合结果时，明确区分部分失败

### 3.6 首页必需行为

首页必须：
- 让任务入口优先于诊断信息
- 清晰展示 active cluster
- 提供直接进入 `Workloads` 的入口
- 不让技术来源分类成为主 UX 语言

### 3.7 Workloads 页必需行为

Workloads 页面必须：
- 清晰展示当前 cluster 和 namespace 范围
- 提供真实资源列表
- 明显暴露 `Create` 或 `Apply`
- 在操作后保留上下文

## 4. V1 可做但非必需

以下内容有价值，但不是 V1 完成条件：
- 按 cluster 恢复上次使用的 namespace
- 首页“最近任务 / 恢复任务”快捷入口
- 轻量空状态引导
- 次级刷新状态提示
- UI 层延后支持 `multiple` namespace 选择

只有在不拖慢首条可用工作流的前提下，才应加入这些内容。

## 5. V1 明确不做

V1 不应扩展到：
- AI 增强工作流
- 通用聊天或助手界面
- 大范围插件驱动工作流交付
- 大型仪表盘式可观测页面
- 超出首条真实工作流之外的大范围资源域扩展
- 在工作流未成立前进行大量视觉打磨
- 对首条任务帮助不大的高级筛选和排序

## 6. V1 后延期方向

以下方向可以在 V1 之后推进：
- 作为后续任务增强层的 AI
- 更广泛的插件承载工作流
- 更丰富的 workload 详情视图
- 完整 UI 形态的 `multiple` namespace 选择
- 超出 `Workloads` 的更多资源域
- 首页任务恢复和任务记忆能力

## 7. 完成检查

当下面所有问题都能回答“是”时，V1 才算在边界内完成：
- 用户能否从首页直接进入 `Workloads`？
- 用户能否立刻理解当前 cluster 和 namespace 上下文？
- 用户能否完成一次真实的 create/apply 操作？
- 这次操作能否收敛到合法且明确的目标？
- 用户能否在不离开任务路径的前提下理解成功与失败？
- 用户能否在操作后继续工作而不丢失上下文？

如果其中任意一个答案是否定的，就不应继续扩大 V1 范围。
