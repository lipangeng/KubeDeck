# KubeDeck V1 实现边界

## 1. 目的

本文档定义在菜单组合系统与资源页扩展模型进入主架构之后，V1 当前必须包含什么。

只有当一条真实工作流在新运行时形状之上可用时，V1 才算完成。

## 2. V1 核心原则

V1 必须同时证明产品价值与架构有效性。

这意味着，以下情况都不能算 V1 完成：

- 壳层能渲染，
- 插件合同存在，
- 菜单和页面技术上可注册。

只有当一条真实工作流通过当前微内核、菜单模型和资源页模型交付出来，V1 才算完成。

## 3. V1 必须交付

### 3.1 运行时基础

V1 必须保留：

- 微内核与插件运行时，
- 组合式菜单行为，
- 统一资源页外壳模型。

### 3.2 首条真实工作流

V1 必须交付一条完整工作流：

1. 进入首页，
2. 确认 active cluster，
3. 通过组合式菜单进入 `Workloads`，
4. 查看真实 workload 数据，
5. 进入资源页或动作入口，
6. 完成一次标准操作，
7. 获得清晰结果反馈，
8. 在不丢失核心上下文的前提下继续工作。

### 3.3 必需页面与表面

V1 必须包含：

- Homepage，
- 由组合式菜单驱动的左侧导航，
- 作为首个真实域的 `Workloads`，
- 一个共享资源页外壳，
- `Overview` 与 `YAML` 两个默认 tabs，
- 以及一条标准动作路径，例如 create、apply 或 edit。

### 3.4 必需上下文

V1 必须保留：

- active cluster，
- namespace scope，
- 当前 menu / workflow domain，
- 进入资源页后的当前资源身份，
- 以及返回上一工作路径所需的连续性状态。

### 3.5 必需 namespace 行为

V1 必须支持：

- `single` namespace scope，
- 浏览场景下的 `all` namespace scope，
- 对需要具体 namespace 的操作进行明确目标解析，
- 在菜单导航、列表页和动作之间保持 namespace 连续性。

### 3.6 必需菜单行为

V1 必须以产品形态体现新的菜单模型：

- 系统拥有的默认菜单骨架，
- 明确的 built-in 入口，
- 为 CRD 与 plugin mounts 预留位置，
- 对“已配置但当前不可用”的菜单项显示禁用态，
- 提供 `CRDs` 兜底资源入口。

V1 可以暂时只做很轻量的用户覆盖存储，但运行时形状不能阻碍后续这层能力。

### 3.7 必需资源页行为

V1 必须以产品形态体现新的资源页模型：

- 第一批资源都通过共享 shell 解析，
- `Overview` 与 `YAML` 是真实 tabs，
- built-in 特化仍然落在同一模型内，
- 并为后续的 tab replacement 与 takeover 预留空间。

## 4. 低成本可选项

以下内容有价值，但不是 V1 完成条件：

- 轻量的全局菜单偏好，
- 带最小持久化的 cluster 局部菜单覆盖，
- 超过一个的特化资源页，
- 多于一个的 YAML 变体，
- Homepage 恢复入口，
- UI 中延后的 `multiple` namespace 选择。

## 5. V1 明确不做

V1 不应扩展到：

- AI 辅助工作流，
- 通用聊天表面，
- 插件市场或热安装 UI，
- 大范围可观测性仪表盘，
- 广泛的块级页面扩展系统，
- 在首个域稳定前扩展大量资源域，
- 在工作流有效性未被证明前做大量视觉打磨。

## 6. V1 完成检查

只有当以下问题全部回答“是”，V1 才算在边界内：

- 这条工作流是否运行在当前微内核运行时之上？
- 左侧菜单是否通过组合生成，而不是硬编码？
- 用户能否从该菜单进入 `Workloads`？
- 用户能否到达真实资源列表与资源页？
- 资源页是否通过统一模型暴露 `Overview` 与 `YAML`？
- 用户能否完成一次真实操作并完成目标解析？
- 用户能否理解结果并在不丢失上下文的前提下继续工作？

只要其中任何一个问题答案为“否”，V1 范围就不应继续扩张。
