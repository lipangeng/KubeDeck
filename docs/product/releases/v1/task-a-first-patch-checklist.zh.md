# KubeDeck Task A 首个补丁检查清单

## 1. 目的

本文档定义 Task A：共享工作上下文模型 的首个补丁在开始前与完成后应检查的最后清单。

它的作用是防止首个补丁偏离其“结构性补丁”的目标。

## 2. 开始前检查清单

只有当以下条件全部成立时，才应开始实现：

### 范围检查

- [ ] 团队已接受：首个补丁是结构性补丁，而不是 UI 完整补丁。
- [ ] 团队已接受：Homepage 与 Workloads 的页面迁移不属于这个补丁。
- [ ] 团队已接受：旧代码可以暂时并存。
- [ ] 团队也已接受：旧代码不是设计事实来源。

### 模型检查

- [ ] 共享工作上下文模型已被接受。
- [ ] 共享工作上下文状态结构已被接受。
- [ ] 事件与更新规则已被接受。
- [ ] 实现映射已被接受。
- [ ] 文件职责草案已被接受。

### 边界检查

- [ ] V1 边界仍然排除 AI。
- [ ] V1 边界仍然排除大范围插件工作流扩张。
- [ ] 首个补丁仍然排除 UI 重写与视觉重设计。

### 补丁范围检查

- [ ] 首个补丁的文件集合已经固定。
- [ ] 首个补丁会先创建新的 `state/work-context` 文件。
- [ ] 首个补丁不会先删除历史页面/状态代码。
- [ ] 首个补丁的测试范围仅限状态规则与事件规则。

## 3. 进行中约束

在首个补丁开发过程中，以下条件必须持续成立：

- [ ] 没有加入 Homepage 视觉重做。
- [ ] 没有加入 Workloads 页面重写。
- [ ] 没有加入 Create / Apply UI 工作。
- [ ] 没有加入插件系统工作。
- [ ] 没有加入后端 API 重设计。
- [ ] 没有把 AI 范围带入补丁。
- [ ] 在替代结构存在前，没有删除旧文件。

如果其中任意一项变成否，说明补丁范围已经漂移。

## 4. 必需交付物

首个补丁必须交付：

- [ ] 新的 `frontend/src/state/work-context/` 文件结构
- [ ] 根共享状态类型
- [ ] cluster 上下文逻辑
- [ ] namespace 上下文逻辑
- [ ] workflow 上下文逻辑
- [ ] action 上下文逻辑
- [ ] list 上下文结构
- [ ] 事件 / 更新入口
- [ ] 面向后续页面消费的 selectors
- [ ] 覆盖核心状态规则的单元测试

## 5. 必需验证

补丁完成后，至少应确认：

- [ ] cluster 切换状态能够被干净表达
- [ ] namespace scope 模式能够被干净表达
- [ ] execution target 在结构上与 browsing scope 分离
- [ ] 事件处理器不会改写无关的连续性状态
- [ ] 共享上下文文件可编译且测试通过

## 6. 漂移信号

如果发生以下任一情况，则说明首个补丁已经偏离方向：

- [ ] `App.tsx` 又变成新模型的事实定义中心
- [ ] namespace 浏览范围和 execution target 被混成一个概念
- [ ] UI 关注点主导了新的状态文件
- [ ] 在共享结构稳定前就开始页面迁移
- [ ] 在替代路径明确前就开始清理旧代码

## 7. 完成标准

当满足以下条件时，首个补丁才算完成：

- [ ] 新的结构性所有权已经建立
- [ ] 核心共享上下文概念已有规范归属
- [ ] 补丁仍然保持在首个补丁范围内
- [ ] 没有混入意外的 UI 迁移
- [ ] 仓库已经准备好进入第二个补丁：页面接入新上下文基础

## 8. 下一补丁启动门槛

只有满足以下条件，下一补丁才应开始：

- [ ] 首个补丁已经合并，或至少已被接受为规范基础
- [ ] 共享上下文文件集合已不再存在争议
- [ ] 事件 / 更新入口的归属已不再存在争议
- [ ] 团队已同意：下一补丁是受控的页面接线，而不是继续重设计状态模型
