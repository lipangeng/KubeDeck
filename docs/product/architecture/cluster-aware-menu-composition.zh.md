# 集群感知菜单组合设计

## 状态

本草案已在需求讨论阶段获得确认。

本文档定义 KubeDeck 在微内核基线之后的目标菜单模型。

## 1. 问题定义

当前内核贡献链已经支持 `page`、`menu`、`action`、`slot` 四类贡献，但左侧导航本质上仍然是“已注册菜单项的渲染结果”。

这还不足以承载目标产品行为。

KubeDeck 需要的菜单系统应该满足：

- 系统提供一套强默认菜单；
- CRD 与插件路由被当成统一的可挂载能力；
- 当前集群能力只改变已配置菜单项的可用状态；
- 用户可以通过全局和集群级覆盖来定制最终菜单。

因此，菜单系统必须从“注册菜单项”升级为“组合菜单结果”。

## 2. 设计目标

菜单系统必须满足以下目标：

1. 保持稳定的默认信息架构。
2. 将内置工作流、CRD、插件路由统一看作可挂载输入。
3. 菜单聚焦高价值工作流，而不是退化成原始 Kubernetes 资源树。
4. 始终提供标准的 `CRDs` 资源兜底入口。
5. 对于已配置但当前不可用的能力，使用禁用态而不是静默删除。
6. 支持用户通过全局和集群级覆盖进行定制。

## 3. 非目标

本设计不打算：

- 将所有发现到的 CRD 全量自动放入主菜单；
- 允许用户重定义能力真相或集群可用性；
- 让菜单配置演变成脚本系统；
- 或用纯 UI 规则替代后端能力合同。

## 4. 核心模型

菜单系统由四层组成。

### 4.1 `MenuBlueprint`

系统拥有的默认菜单骨架。

这一层定义：

- 默认菜单组；
- 菜单组顺序；
- 默认挂载项；
- 基线信息架构。

它本身不直接表示集群真相。

### 4.2 `MenuMount`

可以被挂载到骨架中的能力项。

无论来源如何，所有挂载项都应遵循统一结构。

合法来源包括：

- 内置工作流；
- CRD 驱动的工作流入口；
- 插件路由。

每个挂载项至少应提供：

- `id`
- `sourceType`
- `target`
- `defaultGroup`
- `order`
- `title`
- `icon`
- `visibility`
- `availability constraints`

### 4.3 `MenuCompositionResult`

某个用户、某个集群上下文下最终组合出的菜单结果。

前端导航渲染器只应消费这一层。

### 4.4 `MenuOverride`

作用在组合结果之上的用户定制层。

需要支持两种覆盖范围：

- 用户全局覆盖
- 集群局部覆盖

## 5. 默认菜单骨架

默认一级菜单建议使用四个 group。

### 5.1 `Core`

高频、基础、最能代表主任务的入口。

典型示例：

- `Workloads`
- `Services`
- `Config`
- `Secrets`

### 5.2 `Platform`

偏平台能力或中频资源域。

典型示例：

- `Networking`
- `Storage`
- `Observability`
- 内置平台工作流

### 5.3 `Extensions`

显式挂载进菜单的 CRD 能力和插件路由。

这是 CRD 与插件能力被统一对待的主区域。

### 5.4 `Resources`

资源兜底入口。

这一组不是主任务导航，而是保证完整资源访问路径始终存在。

至少应包含：

- `CRDs`

## 6. CRD 处理规则

CRD 是 Kubernetes 资源的一种，必须始终有统一兜底入口：

- `Resources -> CRDs`

该入口打开：

1. CRD 定义列表；
2. 再进入选中 CRD 的实例列表。

关键规则：

- 菜单系统不负责把所有发现到的 CRD 自动放进主导航。

相反：

- 只有显式挂载进 blueprint 或用户覆盖层的 CRD，才会出现在其他 group 中。

如果某个已配置 CRD 在当前集群中不存在：

- 菜单项仍然显示；
- 但处于禁用状态；
- 并显示可用性图标与提示信息。

## 7. 插件路由处理规则

插件路由与 CRD 菜单入口遵循同一挂载模型。

插件路由：

- 不应天然拥有最终菜单位置；
- 只提供可挂载能力；
- 最终位置由 blueprint 与 override 系统决定。

如果插件能力在当前上下文中不可用：

- 若它是显式配置项，则仍保留显示；
- 但表现为禁用态；
- 除非策略要求隐藏，否则不能静默消失。

## 8. 可用性状态

每个最终菜单项都必须解析为以下三种状态之一：

- `enabled`
- `disabled-unavailable`
- `hidden`

### `enabled`

当前上下文下能力存在且可访问。

### `disabled-unavailable`

菜单项是有意配置的，但当前 cluster 或上下文中缺失对应能力。

这样做是为了保持菜单结构稳定。

### `hidden`

被策略或用户覆盖隐藏。

## 9. 用户覆盖模型

需要两级覆盖。

### 9.1 全局覆盖

对某个用户的所有 cluster 生效。

允许的操作：

- 调整 group 顺序；
- 调整菜单项顺序；
- 在 group 间移动菜单项；
- 隐藏菜单项；
- 固定菜单项；
- 可选地提供展示别名。

### 9.2 集群覆盖

仅对某个 cluster 生效。

适用于：

- 集群特定的排序；
- 集群特定的隐藏；
- 集群特定的位置移动；
- 集群特定的固定操作。

最终组合顺序必须是：

1. 系统 blueprint
2. 动态挂载解析
3. 全局覆盖
4. 集群覆盖

## 10. 护栏

覆盖系统不应允许用户：

- 凭空创造不存在的能力；
- 将不可用能力强行变成可用；
- 重定义后端能力真相；
- 或绕过授权和策略控制。

用户覆盖层只负责菜单组合，不负责能力定义。

## 11. 对当前架构的影响

当前系统已经具备一些有价值的基础：

- backend manifest 发现；
- frontend module 发现；
- `page/menu/action/slot` 贡献合同；
- 微内核端到端组合链路。

但当前 `menu contribution` 的抽象仍然太接近“页面入口声明”。

要支持本文档中的菜单模型，菜单系统需要从：

- “注册菜单项”

升级为：

- “菜单骨架 + 挂载组合”

这不是简单的视觉重构，而是抽象层级的升级。

## 12. 与当前计划的一致性

项目当前仍然处于已确认的微内核路线之内。

最近完成的工作主要是扩展管线本身：

- backend manifest 加载；
- frontend 插件模块发现；
- sample plugin 验证；
- remote page 与 slot 渲染。

当前真正缺少的是本文档描述的更高一层菜单组合系统。

因此项目并没有跑偏，只是菜单编排这一层现在被正式显性化了。

## 13. 推荐的下一步规划

在开始实现菜单系统前，应先单独补一份实现规划，覆盖：

- `MenuBlueprint` schema
- `MenuMount` schema
- 组合流水线
- 用户覆盖持久化模型
- 可用性解析规则

在该规划完成并评审前，不应直接开始菜单实现。
