# KubeDeck 内核贡献合同草案

## 1. 目的

本文档用于定义 KubeDeck 内核在继续推进更大规模功能实现之前，必须先具备的最小贡献合同。

目标是避免内置工作流继续沉淀成永久性的壳层专属逻辑。

## 2. 合同原则

内核不应拥有业务工作流实现本身。

内核应负责：
- 贡献注册，
- 组合规则，
- 运行时上下文边界，
- 以及带权限意识的执行边界。

内置能力与未来插件应尽可能通过同一组合同家族接入。

## 3. 前端合同家族

前端内核必须暴露以下最小贡献家族。

### 3.1 页面贡献合同

目的：
- 注册一个可导航的工作流页面或领域入口

必须支持：
- 贡献身份
- 工作流域身份
- route 或 entry key
- 展示标签元数据
- 页面组件的归属关系

不能默认：
- 页面永远由壳层硬编码
- 文案永远只有一个固定 locale 字符串

### 3.2 菜单贡献合同

目的：
- 向内核组合后的导航结构中贡献一个或多个入口

必须支持：
- 贡献身份
- 菜单放置元数据
- 顺序与可见性元数据
- 可本地化标签元数据
- 与页面或工作流入口的映射关系

不能默认：
- 只有内置菜单来源存在
- 菜单分组永久绑定技术来源分类

### 3.3 动作贡献合同

目的：
- 向 `Workloads` 这类工作流域中贡献可执行动作

必须支持：
- 动作身份
- 所属工作流域
- 展示标签元数据
- 承载界面类型元数据
- 基于后端状态的权限可见性提示

不能默认：
- 动作执行永远直接定义在页面组件内部

### 3.4 扩展槽位贡献合同

目的：
- 允许能力模块向现有工作流页面挂接附加面板、摘要区或上下文区块

必须支持：
- 槽位身份
- 放置元数据
- 贡献归属身份
- 可见条件

V1 不要求：
- 完整 replacement 语义

## 4. 后端合同家族

后端内核必须暴露以下最小能力家族。

### 4.1 插件身份合同

目的：
- 唯一且稳定地标识一个后端能力提供者

必须支持：
- 稳定的插件或能力 ID
- 可版本化的归属边界

这是最薄的一层合同，但它本身并不足够。

### 4.2 能力注册合同

目的：
- 注册一个内置能力模块或未来插件究竟提供了什么

必须支持：
- 页面或工作流能力元数据
- 资源域能力元数据
- 动作能力元数据
- 菜单暴露元数据

### 4.3 元数据暴露合同

目的：
- 向前端壳层暴露内核组合后的菜单、页面和动作元数据

必须支持：
- cluster 感知的元数据解析
- 基于权限的可见性过滤
- 可本地化展示 key 或标签元数据

### 4.4 执行入口合同

目的：
- 通过后端权威策略边界执行用户可见动作

必须支持：
- 动作身份
- 目标上下文
- 权限校验
- 结果摘要结构

不能默认：
- 前端拥有最终执行权威

## 5. 内置能力规则

像 `Homepage`、`Workloads`、`Create`、`Apply` 这类内置工作流域，虽然可以随主仓库一起交付，但其建模方式仍应是“能力贡献”，而不是永久性的壳层例外逻辑。

这在实践上意味着：
- 壳层可以负责引导启动它们，
- 但壳层不应永远内联定义它们的工作流语义。

## 6. 可本地化元数据规则

所有承载用户可见标签的合同家族，都必须允许未来可本地化表示。

至少，合同设计现在就要能容纳：
- label key，
- description key，
- 或未来可按 locale 解析的结构化 label 字段。

V1 不需要先把完整运行时翻译系统做完，但合同形状不能阻塞这一点。

## 7. V1 最小合同集合

在继续扩大实现前，项目至少应先定义：
- 一个页面贡献结构
- 一个菜单贡献结构
- 一个动作贡献结构
- 一个槽位贡献结构
- 一个后端能力注册结构
- 一个后端元数据暴露结构
- 一个后端动作执行结构

这就是最小内核合同基线。

## 8. 可以后置的合同深度

以下内容可以在首条工作流稳定后再做：
- 插件依赖解析规则
- 动态安装与卸载流程
- 丰富的版本协商机制
- 多贡献者间的 slot replacement 优先级
- 插件沙箱细节策略

这些是合理的未来问题，但不是当前建立合同家族所必需的内容。

## 9. 就绪检查

当以下问题都可以回答“是”时，这份合同草案才算可用：
- `Homepage` 与 `Workloads` 是否可以被描述为页面贡献？
- `Create` 与 `Apply` 是否可以被描述为动作贡献？
- 导航是否可以被描述为菜单贡献，而不是壳层自有列表？
- 未来侧边面板或摘要区是否可以被描述为槽位贡献？
- 后端执行是否可以通过显式动作合同保持权威？
