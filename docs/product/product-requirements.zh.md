# KubeDeck 产品需求澄清

## 1. 目的

本文档用于澄清 KubeDeck 的原始产品需求，确保后续的 UI、架构和实现决策都围绕同一个目标展开。

## 2. 问题定义

在多 Kubernetes 集群环境中，团队往往依赖碎片化工具链：
- 使用 `kubectl` 进行底层操作，
- 使用独立控制台查看集群状态，
- 使用临时内部工具承载团队自定义流程。

这种分裂会带来频繁的上下文切换、一致性下降，以及更高的自定义流程维护成本。

## 3. 产品目标

KubeDeck 应提供一个统一且可扩展的 Web 控制平面，使用户能够：
- 在清晰的多集群上下文中工作，
- 以统一模型处理内置资源与 CRD，
- 并通过插件访问团队特有的工作流。

## 4. 目标用户

主要目标用户包括：
- 平台工程师，
- SRE / DevOps 运维人员，
- 跨多个集群交付业务的工程角色。

后续可以扩展到其他角色，但产品初期不应优先为只读或低频用户优化。

## 5. 核心用户任务

首批必须支持的核心任务包括：
- 选择正确的 cluster，
- 进入高频资源域或工作流入口，
- 查看相关资源，
- 执行一次标准操作，
- 并在不丢失上下文的情况下继续后续工作。

## 6. 独特价值

KubeDeck 的差异化价值应由以下能力共同构成：
- 将多集群上下文作为第一公民，
- 统一内置资源与 CRD 的处理方式，
- 通过插件扩展团队工作流，
- 并由后端负责权威访问控制。

它的价值不应该只是“又一个 Kubernetes 控制台”。

## 7. AI 产品定位

KubeDeck 中的 AI 应被视为后续的任务增强层，而不是 MVP 组成部分，也不是一个孤立的聊天功能。

AI 应利用当前 cluster、namespace、resource、page 和 workflow 上下文，帮助用户完成真实的控制面任务。

但 AI 只能在第一条非 AI 的基础任务流端到端成立后再引入。在此之前引入 AI，只会增加复杂度，而无法真正放大产品价值。

首批最值得引入 AI 的三条任务流是：
- 资源创建与变更执行，
- 异常排查与状态理解，
- 以及插件承载的团队工作流。

在这些流程中，AI 最适合扮演：
- 意图翻译器，
- YAML 与操作辅助器，
- 风险与结果解释器，
- 状态摘要器，
- 以及嵌入插件体验中的工作流助手。

AI 不应被引入为：
- 通用首页聊天框，
- 脱离上下文的 Kubernetes 问答框，
- 或绕过权限与工作流控制的执行入口。

## 8. 最小可用闭环

第一个必须端到端成立的工作流是：

1. 进入系统。
2. 选择或确认 cluster。
3. 进入一个具体资源域。
4. 看到真实资源。
5. 执行一次标准操作，例如 create 或 apply。
6. 获得可执行的结果反馈。
7. 在同一工作上下文中继续操作。

AI 未来可以增强这条闭环，例如帮助用户起草变更、校验意图、解释结果，但 AI 不应替代这条闭环本身，也不是 MVP 必需项。

## 9. 首页需求

首页应帮助用户：
- 明确当前 cluster 上下文，
- 找到下一步最可能的任务，
- 并快速进入主工作流。

首页不应主要为框架诊断、开发验证或元数据检查服务。

如果首页出现 AI，它也只能帮助用户进入或继续当前上下文中的任务，而不应作为独立功能主导首页。

## 10. 当前 UI 为什么偏离目标

当前 UI 偏离产品目标，主要因为：
- 它优先强调运行状态和壳层是否就绪，而不是任务入口，
- 它按导航来源分类，而不是按用户任务分类，
- 它尚未把用户带入一条完整工作流。

因此，这个界面目前更清楚地表达了实现进度，而不是产品价值。

这也意味着现在不能把 AI 作为“表面创新”直接叠加到首页结构上。若没有真实任务流，AI 只会放大错误的产品重心，并提前扩大范围。

## 11. 决策判断标准

后续变更应优先用以下问题判断：
- 这是否让第一条用户任务流更完整？
- 这是否减少了多集群上下文切换成本？
- 这是否强化了统一资源模型？
- 这是否支持有意义的插件扩展？
- 这是否比“展示框架状态”更能提升用户任务完成度？
