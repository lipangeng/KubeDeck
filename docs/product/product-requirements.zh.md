# KubeDeck 产品需求

## 1. 目的

本文档是 KubeDeck 当前有效的产品需求基线。

它定义产品要解决什么问题、服务哪些用户、首版必须证明什么价值，以及哪些方向当前明确后置。

## 2. 产品问题

多集群 Kubernetes 团队通常需要在多种工具之间切换：

- 用 `kubectl` 做底层变更，
- 用独立面板查看状态，
- 用内部工具承载团队工作流，
- 用零散文档理解 CRD 与自定义操作。

这种割裂会带来：

- 操作上下文频繁切换，
- 工作路径不一致，
- 定制工作流成本高，
- 跨集群与跨资源类型的连续性差。

## 3. 产品目标

KubeDeck 应提供一个可扩展的 Web 控制面，让用户能够：

- 在明确的 cluster 上下文中操作，
- 用统一模型处理内置资源与 CRD，
- 通过面向任务的菜单进入工作，而不是面对原始资源堆叠，
- 通过插件能力扩展团队工作流。

## 4. 目标用户

首要目标用户包括：

- 平台工程师，
- SRE / DevOps，
- 需要跨多个 cluster 交付和维护资源的工程师。

产品初期不以只读用户或纯仪表盘场景为主要优化对象。

## 5. 核心用户任务

第一批必须支持的任务是：

1. 确认正确的 cluster 与工作范围，
2. 快速进入真实资源域或工作流域，
3. 在稳定上下文中查看资源，
4. 完成一次标准操作，例如 create、apply 或 edit，
5. 在不丢失 cluster / namespace 连续性的前提下继续后续工作。

## 6. 产品独特价值

KubeDeck 的差异化价值来自：

- 将多集群上下文作为一等公民，
- 用统一心智模型承载内置资源与 CRD，
- 通过微内核与插件运行时承载扩展能力，
- 用声明式菜单组合代替硬编码导航，
- 用统一资源页外壳支持默认能力与渐进式特化。

它的价值不是“再做一个 Kubernetes 仪表盘”。

## 7. 导航需求

导航必须采用组合式菜单模型，而不是静态渲染已注册入口列表。

菜单系统必须提供：

- 系统拥有的默认菜单骨架，
- 对内置工作流、CRD 入口、插件路由的显式挂载，
- 基于 cluster 的可用性解析，
- 标准的 `CRDs` 兜底资源入口，
- 以及全局偏好与 cluster 局部覆盖两层用户自定义。

只有显式配置进菜单骨架的 CRD，才应出现在业务菜单组中。

如果某个已配置的 CRD 在当前 cluster 不存在，菜单项应保留但禁用，并给出明确提示。

## 8. 资源页面需求

资源页必须遵循统一模型。

产品应采用：

- 统一的 `ResourcePageShell`，
- 默认的 `Overview` 与 `YAML` tabs，
- 近期以 tab-first 方式扩展，
- 并在必要时支持 tab 替换或整页接管。

CRD 必须始终具备安全的默认访问路径，其中包含 YAML 能力。

像 `Pod` 这样的内置资源未来可以演进出更复杂的体验，但仍应从同一顶层页面模型出发。

## 9. MVP 定义

V1 必须在新架构之上证明一条真实工作流。

最小产品闭环是：

1. 用户进入系统，
2. 确认 cluster 上下文，
3. 进入真实资源域，例如 `Workloads`，
4. 查看真实资源，
5. 完成一次标准操作，
6. 获得清晰结果反馈，
7. 保持当前工作上下文继续下一步。

如果这条闭环未完成，产品仍然只是一个可扩展外壳，而不是可用控制面。

## 10. 首页需求

首页必须帮助用户：

- 确认当前上下文，
- 看到最可能的下一步任务，
- 并立即进入该任务。

首页不能主要表现为：

- 运行时诊断面板，
- 框架验证界面，
- 或原始元数据展示页。

## 11. 当前非目标

以下方向当前明确后置：

- AI 辅助工作流，
- 通用聊天界面，
- 完整 i18n 产品交付，
- 大型可观测性仪表盘，
- 在首条工作流稳定前扩展大量资源域，
- 细粒度块级资源页扩展。

## 12. 决策过滤器

后续产品与实现决策都应通过以下问题判断：

- 这是否强化了首条真实工作流？
- 这是否保持了微内核与插件运行时的形状？
- 这是否符合组合式菜单模型？
- 这是否符合统一资源页外壳模型？
- 这是否减少了跨 cluster 与跨资源类型的上下文切换？
- 这是否带来的产品价值大于它增加的框架复杂度？
