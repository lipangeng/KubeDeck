# KubeDeck 首个基础任务流草案

## 1. 目的

本文档定义 KubeDeck 必须优先成立的第一条核心任务流。在这条链路打通之前，不应优先扩大 UI 范围、插件范围或 AI 增强范围。

## 2. 目标用户

首个任务流应优先服务：
- 平台工程师，
- SRE / DevOps 运维人员，
- 以及需要跨多个集群执行真实操作的交付工程角色。

这条流不应优先面向被动查看者或低频用户设计。

## 3. 任务目标

首个任务流的目标是：
- 进入系统，
- 确认正确的 cluster 上下文，
- 进入一个真实资源域，
- 执行一次标准资源操作，
- 获得明确结果，
- 并在不丢失上下文的情况下继续工作。

简而言之，这条任务流必须证明 KubeDeck 是一个可用控制平面，而不只是一个可展示的壳。

## 4. 任务流定义

首个核心任务流为：

1. 进入首页。
2. 确认或切换当前 cluster。
3. 进入 `Workloads`。
4. 查看真实的 workload 列表。
5. 发起一个标准操作，例如 `Create` 或 `Apply`。
6. 提交操作。
7. 获得成功、部分失败或失败反馈。
8. 返回 workload 上下文并继续工作。

## 5. 页面清单

最小页面集合应包括：
- 首页
- Workloads 页面
- Create / Apply 操作界面
- 同一任务流内的结果反馈状态

Create / Apply 操作界面可以是页面、抽屉或弹窗。对 MVP 而言，关键是任务连续性，而不是具体容器形式。

## 6. 每页唯一职责

### 首页

唯一职责：
- 建立工作上下文，
- 并把用户快速送入主任务流。

首页不应成为诊断信息、框架元数据或实现进度展示的主要承载面。

### Workloads 页面

唯一职责：
- 作为当前上下文中浏览和选择资源的主工作面。

这个页面必须让 cluster、namespace 和资源范围一目了然。

### Create / Apply 操作界面

唯一职责：
- 收集输入并完成一次标准操作。

它不应迫使用户重新建立上下文，也不应让用户为了无关信息跳转到别处。

### 结果反馈状态

唯一职责：
- 解释提交后发生了什么，
- 展示哪些成功、哪些失败，
- 并引导用户进行下一步操作。

## 7. 页面流转

预期页面流转如下：

1. 首页
2. Workloads 页面
3. Create / Apply 操作界面
4. 提交状态
5. 结果反馈
6. 保留上下文返回 Workloads 页面

如果用户在提交过程中失败，流程也应停留在同一任务路径内，而不是把用户打回首页。

## 8. 关键状态

这条任务流必须稳定保留以下状态：
- `activeCluster`
- `namespace`
- 当前资源域
- 当前列表筛选条件
- 当前操作模式
- 提交状态
- 操作结果摘要

其中最关键的两个锚点是 `activeCluster` 和 `namespace`。

## 9. 成功反馈规则

成功反馈必须：
- 明确告诉用户操作成功，
- 明确指出哪些对象受到了影响，
- 保留用户当前上下文，
- 并允许用户立刻继续操作。

成功反馈不应短暂消失后在当前工作视图中不留下任何痕迹。

## 10. 失败反馈规则

失败反馈必须：
- 明确告诉用户操作失败，
- 明确指出失败发生在哪一步或哪个对象上，
- 保留用户输入上下文，
- 并告诉用户应该重试、修改还是返回。

如果存在部分失败，成功项和失败项必须可区分。

失败反馈不应只是一段原始后端错误字符串。

## 11. 首页信息架构草案

首页应保留：
- 当前 cluster 选择器，
- 主任务入口，
- 当前工作上下文摘要，
- 以及少量真正阻塞用户继续工作的状态提示。

首页应降级或移除出主视觉焦点：
- 作为主内容展示的运行时健康状态，
- 原始 API target 信息，
- registry 类型计数，
- 面向壳层诊断的失败摘要，
- 按实现来源分组的导航。

首页第一优先回答的应是这三个问题：
- 我现在在哪个 cluster？
- 我接下来最可能要做什么？
- 我能不能立刻进入这个任务？

## 12. 范围边界

这条首个任务流不要求：
- 在任务中引入 AI，
- 完整插件生态，
- 大范围资源覆盖，
- 或完整控制面仪表盘。

它只要求一条真实、可重复、任务闭环完整的操作路径。

## 13. 页面模块清单

### 首页模块

必需模块：
- 全局头部
- 当前 cluster 选择器
- 主任务入口卡片
- 当前上下文摘要
- 最近或默认任务快捷入口
- 阻塞性状态提示区域

可选但低优先级模块：
- 轻量通知
- 最近活动摘要

首页不应把诊断类小组件作为主模块。

### Workloads 页面模块

必需模块：
- 展示当前 cluster 和 namespace 的页面头部
- 资源域导航或标题
- namespace 选择器或 namespace 范围展示
- 过滤与搜索区域
- workload 列表或表格
- `Create` 或 `Apply` 等主操作入口
- 刷新或重新加载控件

可选但低优先级模块：
- 空状态引导
- 次级摘要指标

### Create / Apply 操作界面模块

必需模块：
- 操作标题和当前上下文摘要
- namespace 确认字段
- YAML 或结构化表单输入区
- 校验与警告区域
- 提交操作
- 取消 / 返回操作

可选但低优先级模块：
- 示例或模板
- 高级选项区域

### 结果反馈模块

必需模块：
- 结果状态标题
- 受影响对象摘要
- 成功 / 失败 / 部分失败拆分区
- 可执行下一步区域
- 返回 workload 上下文的操作入口

可选但低优先级模块：
- 可展开的原始技术细节
- 可复制的诊断信息载荷

## 14. 状态迁移规则

### 主要工作流状态

首个任务流应使用以下高层状态：
- `homepage_idle`
- `cluster_switching`
- `workloads_loading`
- `workloads_ready`
- `operation_editing`
- `operation_validating`
- `operation_submitting`
- `operation_success`
- `operation_partial_failure`
- `operation_failure`

### 状态迁移规则

1. 进入首页
- 默认状态为 `homepage_idle`
- 如果用户切换 cluster，转入 `cluster_switching`

2. 切换 cluster
- 从 `cluster_switching` 开始加载目标 workload 上下文
- 成功时转入 `workloads_loading`，随后进入 `workloads_ready`
- 失败时保留当前已知安全上下文，并展示阻塞反馈

3. 进入 workload 工作面
- 从首页主任务入口进入时，转入 `workloads_loading`
- 当列表数据和上下文可用后，进入 `workloads_ready`

4. 开始 create/apply
- 从 `workloads_ready` 打开操作界面后，转入 `operation_editing`

5. 校验输入
- 从 `operation_editing` 触发检查或提交前预检查时，转入 `operation_validating`
- 校验成功后，返回 `operation_editing`，并进入可提交状态
- 校验失败后，仍留在 `operation_editing`，并保留用户输入

6. 提交操作
- 从 `operation_editing` 提交后，转入 `operation_submitting`
- 提交过程中，输入内容必须可恢复

7. 提交结果
- 全部成功时，转入 `operation_success`
- 部分成功时，转入 `operation_partial_failure`
- 失败时，转入 `operation_failure`

8. 返回工作面
- 从任一结果状态返回时，转入 `workloads_loading`
- 当刷新后的数据可用时，进入 `workloads_ready`
- 必须保留 cluster、namespace 和相关筛选条件

### 状态保留规则

除非用户主动修改，下列值必须在整个任务流中保留：
- 当前 active cluster
- namespace
- 当前资源域
- 列表筛选状态

下列值可在操作完成后重置：
- 草稿输入，但仅限用户明确退出或开始新操作时
- 临时校验消息

### 错误处理规则

- 提交失败不能重置 cluster 或 namespace 上下文
- 校验失败不能丢弃用户输入
- 提交后若 workload 刷新失败，必须保留结果反馈直到用户确认
- cluster 切换失败不能让用户停留在未定义的混合上下文中
